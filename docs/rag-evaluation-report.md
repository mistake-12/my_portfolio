> 评测时间：2026-06-08  
> 评测框架：RAGAS LLM-as-Judge  
> 测试集：30 条（资料命中 15 + FAQ 5 + 项目边界 5 + 兜底拒绝 5）

## 一、系统架构

```
用户问题
  ├─ FAQ 正则精确匹配？ ──→ 是 → 直接返回预设答案
  └─ 否
      └─ 智谱 embedding-2 向量化
          └─ 混合检索（向量余弦相似度 60% + BM25关键词 40%）
              └─ RRF 融合排序
                  └─ 余弦相似度 0.3 阈值过滤
                      └─ 【来源标注】+ System Prompt 拼接
                          └─ DeepSeek 流式生成
```

| 层面 | 策略 | 文件 |
|------|------|------|
| 切分 | 空行 `\n\n` 分段，500字上限，80字滑动窗口重叠 | `src/lib/rag-index.ts` |
| Embedding | 智谱 GLM `embedding-2`（1024维） | `src/lib/embedding.ts` |
| 检索 | 混合检索：向量（余弦相似度 60%）+ BM25 关键词（40%），RRF 融合 | `src/lib/rag-index.ts` |
| 排序 | RRF 归一化分数降序 | `src/lib/rag-index.ts` |
| 过滤 | 相似度阈值 0.3，低于阈值走兜底 | `src/app/api/rag/chat/route.ts` |
| 生成 | DeepSeek-chat 流式输出，带来源标注和硬边界策略 | `src/app/api/rag/chat/route.ts` |
| FAQ分流 | 正则精确匹配短问句（联系方式、薪资等） | `src/app/api/rag/chat/route.ts` |

## 三、各轮优化详情

### R1 → R2：检索层 + Prompt 基础优化

**问题诊断：** Chunk 无重叠导致关键信息被段落边界切断；裸文本无来源标记导致跨项目混淆；prompt 过于宽松导致 LLM 自由发挥。

**优化措施：**
- **滑动窗口重叠**（`rag-index.ts`）：每个 chunk 尾部 80 字复用为下一个 chunk 前缀
- **来源标注**（`route.ts`）：chunk 前缀 `【来源: resume.md】`
- **Prompt 强化**：添加项目区分规则 + 拒绝示例 + 长度约束

**效果：**
- 项目边界 Faithfulness：7.0 → 8.6（+1.6）
- 兜底拒绝 Faithfulness：2.0 → 6.0（+4.0）
- Answer Relevancy 下降（8.9→8.0）：prompt 过度收紧导致部分回答被拒绝

---

### R2 → R3：知识库补全 + 人格化策略

**问题诊断：** R2 的 prompt 太保守——"缺点是什么？""你的风格？"这类人格化问题也被拒绝，但个人网站场景需要能回答。

**优化措施：**
- **自我认知 KB**（`philosophy.md`）：新增"自我认知"段落，描述性格特质
- **Prompt 策略调整**：区分"关于我的开放问题"（允许合理推导）和"完全无关问题"（拒绝）

**效果：**
- 资料命中 Faithfulness：7.4 → 8.3（+0.9）
- 项目边界 Faithfulness：8.6 → 6.0（-2.6，策略松动导致串扰回升）
- 兜底拒绝 Faithfulness：6.0 → 2.0（-4.0，人格化策略过于宽松）

**教训：** prompt 放松后 LLM 在所有维度上都会更自由——需要更精细的边界划分。

---

### R3 → R4：混合检索

**问题诊断：** 纯向量检索对专有名词（"建筑学"、"Qwen3-30B"）召回不足。

**优化措施：**
- **BM25 关键词检索**（`rag-index.ts`）：独立的 TF-IDF 风格关键词评分
- **RRF 融合**：向量检索（60%权重）+ 关键词检索（40%权重），k=60
- **API 层切换**：`searchChunksWithScores` → `hybridSearch`

**效果：** 整体分数反而下降（F: 7.2→6.9），原因是中文分词 tokenizer 将整句中文当作单一 token，关键词检索返回空结果，RRF 的归一化分数也被阈值 0.3 过滤。

**教训：** 混合检索的核心是分词质量——中文不能用空格分词。

---

### R4 → R5：中文 Bigram 分词 + 知识库补全

**问题诊断：** R4 的关键词检索因为中文分词失效而空跑。同时残留的检索遗漏（"建筑学"未被召回）需要知识库层面配合。

**优化措施：**
- **Bigram 分词**（`rag-index.ts`）：中文用字符级 2-gram + 单字，英文用空格分词
- **教育背景补全**（`resume.md`）："本科 建筑学 2022-2024" → "大一大二在建筑系学习...大二后转入工业设计"
- **RRF 分数归一化**：除以最大值，使分数回到 0-1 范围

**效果：**
- HIT-02 "以前学什么专业"：0 → **10**（bigram + KB补全联合生效）
- 资料命中 F：8.2 → 8.9（+0.7）
- 但边界和 OOS 继续崩溃（prompt 策略仍然太松）

---

### R5 → R6：技术硬边界策略 + 知识库补全

**问题诊断：** prompt 的"人格化回答"策略泄漏到了技术类问题——"你会 React Native 吗？" LLM 编造"我接触过一些"；缺乏按问题类型分级的拒绝边界。

**优化措施：**

```
【技术技能 - 硬边界】
凡问技术问题 → KB中有就答，没有就说"这个我不太会"
禁止说"我接触过一些""我在学习""我可以试试"——这些就是编造

【人格特质 - 软边界】  
凡问性格/缺点/工作方式 → 可以基于自我认知KB合理推导

【项目数据 - 硬边界】
具体数字/模型名 → 必须KB有明确记载
```

- **Qwen3-30B 补全**（`resume.md`）：Cobopi 标题加入"大模型选型为Qwen3-30B本地部署"

**效果：**
- HIT-15 "Cobopi大模型"：0 → **10**
- OOS-01 "React Native"：0 → **10**（正确拒绝）
- OOS-04 "写爬虫"：0 → **10**（正确拒绝）
- OOS-03 "特朗普"：0 → **10**（正确拒绝）
- 资料命中 F：8.9 → **9.7**（+0.8）
- 兜底拒绝 F：0.0 → **6.0**（+6.0）

## 六、Prompt 策略设计原则

| 问题类型 | 边界硬度 | 策略 | 示例 |
|---------|:--:|------|------|
| 技术技能 | 🔴 硬 | KB中有→答，没有→拒绝。禁止模糊表述 | "React Native？" → "这个我不太会" |
| 项目具体数据 | 🔴 硬 | 数字/模型名必须 KB 中明确记载 | "识别率多少？" → "约80%" |
| 性格/优缺点 | 🟡 软 | 基于自我认知 KB 合理推导 | "最大缺点？" → 基于 KB 回答 |
| 工作态度/理念 | 🟡 软 | 基于 philosophy + 项目经历推导 | "设计风格？" → 基于 KB 回答 |
| 完全无关话题 | 🔴 硬 | 直接拒绝 | "特朗普？" → 拒绝 |

## 七、Context Precision 缺失说明

当前无法评测 Context Precision，因为 RAG API（`/api/rag/chat`）只返回 AI 生成的回答文本，不暴露检索层数据。

**加入方案：**
1. **API 暴露检索数据**：在 SSE 首个事件中返回 `{"type": "retrieval", "chunks": [...]}`
2. **LLM 标注**：对每个检索到的 chunk，用 LLM Judge 判断是否与问题相关
3. **计算精度**：`Context Precision = 相关chunk数 / 检索到的总chunk数`
